# v1.0 Spec Freeze & Conformance Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Spec sources: `SPEC.md`, [../spec/manifest-schema.md](../spec/manifest-schema.md), [../spec/readme-template.md](../spec/readme-template.md).

**Goal:** Freeze the required manifest fields forever, publish a conformance suite with test vectors, and ship the first README-template translations. This is the heart of v1.0 — the moment the format stops belonging to this repo.

**Context for a fresh executor:** read `CLAUDE.md`, then `SPEC.md` end to end (it is short by design). The reference implementation is `js/manifest.js` + `js/zip.js` + `js/token.js`, all pure and fixture-tested (`tools/test-*.js`). Every task here is format work: `/spec-sync` after each one, and any fixture change follows the format-event protocol (decisions.md + spec version note, never silent).

---

### Task 1: the freeze audit

- [ ] **Step 1:** Table every manifest field as of now: name · type · required/optional · introduced-in (git history of `docs/spec/manifest-schema.md` tells you). Fields marked required at freeze are **required forever** — v2+ may add optional fields only, never rename, never remove, never change meaning.
- [ ] **Step 2:** Resolve every open "reserved" field: `writeback` (used since v0.2), `encryption` (optional since v0.2), `writers` (optional since v0.3 if co-written shipped), `hint`. Each is either frozen-as-specified or explicitly dropped **before** freeze — a reserved-but-undefined field must not survive into v1.0.
- [ ] **Step 3:** Author sign-off on the freeze list (this is a promise made forever; it is theirs to make). Record in decisions.md.
- [ ] **Step 4:** SPEC.md declares v1.0: required core frozen; the compatibility promise written as a section ("a valid v1.0 letter is a valid letter in every future version").

### Task 2: conformance suite + test vectors

- [ ] **Step 1:** Create `conformance/` (in-repo, published with the site as static files): `vectors/letter-minimal/` and `vectors/letter-full/` — two complete, real Tessera folders (README, letter, manifest, checksums, token.svg) generated by the reference implementation from pinned inputs, committed byte-exact.
- [ ] **Step 2:** `conformance/CHECKLIST.md` — the implementer-facing conformance list, each row testable without this repo's code: folder contents, README section order, LF rule, dates-in-words presence, ID derivation (worked example digit by digit from the pinned input), tokenSeed derivation (same), checksum recomputation, zip store-method rule.
- [ ] **Step 3:** `tools/test-conformance.js` — the reference implementation regenerates both vectors from the pinned inputs and byte-compares against the committed folders (this is the same fixture discipline, promoted to format law). Wire into `/qa` Gate 1.
- [ ] **Step 4:** A static page (`conformance.html` or a section of About) that names the suite, links the vectors, and offers the listing: *implementations that pass may say "reads and writes Tessera letters"; listings are free; letters are never certified, only software.* Author reviews wording (it is the public promise).
- [ ] **Step 5:** `/spec-sync` + `/century-audit` + commit.

### Task 3: README template translations

- [ ] **Step 1 (the hard part, decide first):** dates-in-words per language. `datesInWords(iso, lang)` in `js/manifest.js` already takes a `lang` seam — verify (grep it) — then implement per added language as pure lookup-table prose (no `toLocale*`, ever: locale functions are nondeterministic across environments — the determinism rule outranks convenience). Each language: month names, ordinal words, the date sentence pattern.
- [ ] **Step 2:** `docs/spec/readme-template.<lang>.md` per language, starting Swedish (`sv`) and Chinese (`zh`), each CC0. Translation quality bar: **a native reader reviews each; their name-or-decline goes in decisions.md** (same pattern as the grief set).
- [ ] **Step 3:** The compose flow gains a letter-language choice only when at least one translation ships (default: browser-agnostic explicit choice, remembered in settings — never auto-detected silently; the writer knows their reader).
- [ ] **Step 4:** New fixture set per language (`tools/fixtures/readme-<lang>-fix1.txt`), fixture-birth protocol; `node tools/test-manifest.js` extended per language. Gate 2 prose scan (the rules are language-agnostic in spirit; em-dash and urgency rules apply as typography).
- [ ] **Step 5:** `/spec-sync`, commit per language.

### Task 4: third-party implementer invitation

- [ ] **Step 1:** `SPEC.md` gains the standing invitation section: how to claim conformance (pass the suite), how to get listed (an issue/PR), and the CC0 restatement.
- [ ] **Step 2:** README.md (repo) links spec + conformance page prominently. Author gate 2 (announcement timing/copy) governs *when* this goes loud; the pages themselves can exist quietly first.
