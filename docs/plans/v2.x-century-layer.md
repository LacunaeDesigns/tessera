# v2.x Century Layer — plans and gated definitions

> **For agentic workers:** items 1–3 are executable protocols (use superpowers:executing-plans); items 4–6 are **deliberately not step plans** — their own canon ([../features/century-layer.md](../features/century-layer.md)) says each "arrives with its own feature file, fixtures, and century-test rows before build". A step plan written before those exist would be fiction. Each gated item states exactly what unblocks it.

**Context for a fresh executor:** read `CLAUDE.md`; every item here is independent and optional; nothing enters a release without passing the century test on its own.

## 1. Paper is the backup (research protocol → then a plan)

The claim to make total: digital → paper → digital with no infrastructure. Ships only with committed fixtures and a documented, dependency-free decoder. Run the research as its own arc:

- [ ] **Phase A — encoding candidates:** evaluate public-domain 2D encodings (QR at high EC level in a sheaf; DataMatrix; a documented raw-bitmap grid as the fallback candidate). Criteria table: bytes/A4-sheet at home-printer 600dpi · decode robustness after fold/fade (simulate: print, scan at 300dpi, crumple once, rescan) · **spec availability on paper** (can the decode algorithm itself be printed in a few pages? — this criterion is unusual and decisive).
- [ ] **Phase B — prototype:** encode a real sealed folder (the v0.1 test letter's zip bytes, ~10–20KB) with the leading candidate; target ≤3 A4 sheets. Print, scan, decode with a written-for-this, dependency-free decoder script. The loop must close on a real desk, not in memory.
- [ ] **Phase C — decision:** decisions.md entry with the evidence; only then write the implementation plan (data-sheet renderer in `print.js`, plain-text header naming the encoding and where its spec lives, the instructions-page section explaining the sheets to a finder, `tools/test-databack.js` round-trip fixtures).
- [ ] **Kill criterion (honest):** if no candidate decodes reliably after one crumple on home equipment, the feature waits — a backup that fails when needed is worse than the plain honesty of "the paper letter is the backup".

## 2. Audio letters (format-event plan)

- [ ] **Step 1 (format first):** SPEC: `media/` directory rules — recording as WAV or MP3, `letter.txt` remains canonical (the transcript IS the letter; the recording is the treasure), manifest optional `media: [{ file, kind: "audio", description }]`, checksums cover media files. Format-event protocol: SPEC + schema doc + fixtures + decisions.md, same sitting.
- [ ] **Step 2:** Instructions page grows the century sentence: what the file is, even if nothing nearby can play it (one line, prose rules apply).
- [ ] **Step 3:** Writing room: record via MediaRecorder (browser API, no dependency), require a transcript before sealing (typed or corrected — never sealed transcript-less), preview playback, size honesty ("eleven minutes is about ten megabytes; a folder should still fit on anything").
- [ ] **Step 4:** Opening flow plays enclosed audio when present. /qa both paths; `/century-audit` row: an audio letter with a dead codec is still a complete letter via its transcript.

## 4. Letter exchange — gated

**Not software yet, maybe never (the canon accepts that ending).** Definition of ready: a written trust design that survives "no accounts, no server" — otherwise it runs as a periodic human-run exchange event (Delivery-Day adjacent, physical post). If an event runs twice and wants automation, write the design doc first; the plan comes after the doc, not before.

## 5. Estate pack — gated (author gate 4)
