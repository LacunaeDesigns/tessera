# v1.0 Commons Implementation Plan (Delivery Day · gallery · kits)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Spec: [../features/commons.md](../features/commons.md).

**Goal:** The optional shared layer: Delivery Day page, the curated gallery of opened letters, and four printable kits. All of it severable — deleting the commons must leave tool + format whole (`/century-audit` enforces this).

**Context for a fresh executor:** read `CLAUDE.md`. The commons is static pages + one intake pipe; **no engine changes**. Print machinery for kits: `js/print.js` + the print engine in `style.css`. Cost stance (author, 2026-07-10): $0 — free tiers only; the intake scaling ladder below is already decided.

**Author gates in this plan (do not proceed past them silently):** gallery moderation policy before the gallery opens (gate 5); announcement timing/copy (gate 2); the domain question (main vs. sibling) decided with the announcement plan.

---

### Task 1: Delivery Day

- [ ] **Step 1:** A static page (route `#delivery-day` or `delivery-day.html` — match however About is structured then): what it is (21 December, the winter solstice; June solstice noted for the southern hemisphere), the ritual suggestions from the spec (open together; read aloud only if the reader offers; write back the same evening). One page, house voice, Gate 2 scan.
- [ ] **Step 2:** The countdown: quiet, words not digits (*"Delivery Day is seven weeks away"*), computed client-side, no urgency styling — it must feel like a season, not a sale. `prefers-reduced-motion` trivially satisfied (nothing moves).
- [ ] **Step 3:** The practice ICS: one annual-repeat all-day event via `TesseraIcs.buildIcs` (add `RRULE:FREQ=YEARLY` support to `js/ics.js` — pure change, extend `tools/test-ics.js` fixtures first, fixture-birth protocol).
- [ ] **Step 4:** /qa + commit.

### Task 2: gallery intake (the decided $0 ladder)

- [ ] **Step 1 (launch):** Netlify Form on a submission page. Fields: the letter text as the opener received it · opener's consent affirmation (checkbox, required) · redaction requests · writer-consent affirmation where the writer is reachable · no unopened letters (stated rule, affirmed). Free tier: 100 submissions/month — fine at launch.
- [ ] **Step 2 (documented now, built when volume demands):** a Cloudflare Worker (free tier, 100k req/day) receives the same form POST and files each submission as a GitHub issue via the API; Turnstile (free) for spam. The gallery pipeline reads the issue queue instead of the Netlify inbox; nothing else changes. Keep this step as runnable notes in the repo (`docs/plans/notes-gallery-scaling.md` when built) — do not build it before the cap is actually near.
- [ ] **Step 3:** Consent + minors rule from the spec rendered as the submission page's plain-language terms. Author reviews wording (this is the moderation policy surface — gate 5).

### Task 3: /gallery-triage skill + static entries

- [ ] **Step 1:** Write `.claude/skills/gallery-triage/SKILL.md`: pull submissions (Netlify inbox export or issue queue) → check each against the consent rules mechanically (all affirmations present?) → draft a static entry (letter text, redactions applied, sealed/opened years, occasion — never names unless explicitly permitted) → present drafts to the author. **Nothing publishes automatically; the author approves each entry, every time.**
- [ ] **Step 2:** Entries as static HTML fragments in `gallery/` built into pages by a zero-dependency Node script (`tools/build-gallery.js`, same discipline as serve.js) — or hand-placed while volume is small. Reading pages: one letter per page, print-kit letter typography, ink-light print stylesheet (a gallery letter should print beautifully too).
- [ ] **Step 3:** /qa: gallery pages build statically, print correctly; the intake→triage→publish loop run end-to-end once with a real (own) submission. Commit.

### Task 4: the kits

Each kit is one printable static page (PDF-able via the browser), reusing `print.js` sheet machinery and the print rules; content authored via `/occasion` where prompts are involved. No accounts, no app needed in the room — the paper runs the event.

- [ ] **wedding** — guests write to the couple (opened at 1st/10th/25th anniversary): a table sign, guest cards (4-per-A4, cut hairlines), a basket label, and a one-page host runbook (when to hand out cards, where the box lives, who keeps it).
- [ ] **baby** — shower guests to the child at milestones; same shapes, milestone dates on the cards.
- [ ] **classroom** — write to yourself at graduation: the teacher's one-page runbook is the product (handout, collection, storage in a school that will outlast the teacher's tenure, redistribution logistics); student sheet kept simple enough to photocopy.
- [ ] **graduation/retirement** — colleagues' letters at 5/10 years; card + sign + runbook.
- [ ] Per kit: Gate 2 prose scan; A4 + Letter print check; one real print-and-cut; commit per kit.

### Task 5: severability audit + release

- [ ] `/century-audit` with the commons live: the "known, accepted mortality" row — delete/ignore the commons and confirm tool + format + letters are untouched. `/release` per the house flow; announcement itself waits at author gate 2.
